# ADR-0003：**購物車**記憶體為主並延遲寫入持久化層

**Date**：2026-05-14  
**Status**：Accepted

## Context

市集現場：**購物車**會被連續點按；若每次都同步觸發裝置寫入，在高頻輸入下會增加延遲、耗電與失敗機率。**購物車**又需在意外離開程式時仍可還原，不能完全只存在 Compose 區域記憶體。

## Decision

將進行中之 **購物車** 以 ViewModel（或對應之 coordinator）側的記憶體狀態作為使用者互動來源；在變動後於短延遲內分批寫入裝置本機持久化（debounce）；在程式生命週期退到背景或被回收前則強制 flush，降低「來不及寫檔」的窗口。

## Alternatives considered

- **一切以持久化來源為單一真理** — IO 過於頻繁，對戶外裝置不友善。
- **僅離開時寫一次** — 來電、系統記憶體回收等情境易造成 **購物車**流失；與市集現場耐性不符。

## Consequences

- **較佳操作感**：輸入對裝置寫入有節流；仍保留離開／背景時的一致性努力。
- **較複雜的不變式**：理論上有極短的「記憶體已變更、碟上尚未追上」區間—需依賴 flush 協定與（若需要）對帳欄位的結帳前驗證（見 ADR-0002）。
