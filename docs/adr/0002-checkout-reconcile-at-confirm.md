# ADR-0002：結帳前以「對帳層」驗證畫面應收款與當前小計一致

**Date**：2026-05-14  
**Status**：Accepted

## Context

市集現場：**購物車**可能在極短時間內改變，而應收款又來自鍵盤／折扣等高互動來源；若僅信 UI 側單線計算，容易出現「按下去那一瞬間與伺服器側／狀態側不一致」類型的錯帳風險。同時專案刻意避免複雜多層服務，`ViewModel` 仍須對外給出一條可被測試的結帳驗證路徑。

## Decision

在確認 **結帳** 事件中，強制載入三件互相校驗的事實：當時的目錄小計快照、對應的加價／折讓淨值、以及使用者當下確認所見之**應收款**。將「應收公式」集中到可單獨撰寫單元測試的一層，若不一致即拒絕結帳並提示，不進入持久化更新。

對帳詞彙、金流與細部欄位語意載於：`docs/phase-a-checkout-money-flow.md`。

## Alternatives considered

- **僅在 UI 算出應收款並直呼寫入** — 程式最直白；對非同步競態與複製公式錯誤缺乏防護，錯帳難追查。
- **所有金額僅靠 StateFlow 單鏈組出、禁止 UI 參數** — 可避免重複來源；在現有的 Compose／BottomSheet 互動拆分下改造成本較高，仍以「確認點對帳」務實收口。

## Consequences

- **較強**：對帳意圖與規則**集中**並可單獨加測；未來調整結帳欄位時有明確銜接面。
- **較需注意**：詞彙上多個英文／UI 名目含 `total`，閱讀程式必須搭配詞彙表與階段 A 對照文件，見 `CONTEXT.md` 與 `docs/phase-a-checkout-money-flow.md`。
